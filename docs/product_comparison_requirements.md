# C 端商品对比需求：消费者性价比判断与适配推荐

## 1. 场景定位与 PRD 差异说明

### 1.1 现有 PRD 的定位

现有 `phase1_prd.md` 中的跨平台比价场景服务于**内部运营**（B 端工具）：

- 用户是价格运营和类目运营
- 目标是跨平台价格策略判断
- 输入是商品链接，需要 OpenClaw 采集外部平台页面
- 输出是内部比价报告，供运营决策
- 核心闭环是"采、比、析、审、用"

### 1.2 本需求的定位

本需求服务于**消费者**（C 端功能）：

- 用户是在平台上选购家电的消费者
- 目标是帮消费者判断"这个商品性价比如何"以及"这个商品适不适合我"
- 输入是平台内的商品（不涉及跨平台采集）
- 输出是结构化数据，由前端产品经理决定展示方式
- 核心闭环是"选、比、判、荐"

### 1.3 两者的区别

| 维度 | B 端跨平台比价 | C 端商品对比 |
| --- | --- | --- |
| 使用者 | 运营人员 | 消费者 |
| 数据来源 | 外部平台页面（需采集） | 平台内部商品库 |
| 对比对象 | 同款商品在不同平台的价格和福利 | 同品类不同商品之间的功能、参数、适用性 |
| 核心能力 | 价格标准化、福利拆解 | 语义理解、功能对比、场景适配 |
| 输出形式 | 内部比价报告 | 结构化 JSON 给前端展示 |
| AI 理解深度 | 理解价格规则和优惠口径 | 理解商品功能本质和使用场景 |

### 1.4 两者的关联

- 共用**商品知识底座**：商品属性、卖点、参数都依赖同一套知识 RAG
- 共用**语义理解能力**：B 端的"同款判断"和 C 端的"功能对比"都需要 AI 理解商品本质，而不是字段匹配
- C 端商品对比可以复用 B 端已有的 **Offer Normalization Engine** 对价格和福利做标准化
- B 端的 Price Intelligence Agent 积累的商品对齐逻辑，可以为 C 端的"类似商品推荐"提供基础

### 1.5 "数字员工"定位

本场景下的商品对比 Agent 不是一个聊天机器人，而是一个**幕后的家电产品专员**：

- 它像一个在家电卖场工作十年的老员工，真正理解不同品类产品的功能差异
- 比较洗衣机和烘干机时，它知道这两个品类的核心功能不同，不会把"转速"和"烘干温度"放在一起比
- 比较两台洗衣机时，它知道"变频直驱"和"皮带传动"对噪音和衣物保护的实际影响
- 它的输出不是给消费者看的对话文字，而是给前端的结构化数据，由前端决定怎么展示

## 2. 用户画像

### 2.1 主要用户：家电选购消费者

**基本特征**：

- 在平台上浏览家电商品，正处于"考虑购买"阶段
- 对家电参数有一定认知但不专业（知道"变频"但不清楚具体影响）
- 典型年龄 25-45 岁，有独立购买决策权

**典型场景**：

| 场景 | 描述 | 痛点 |
| --- | --- | --- |
| 两款犹豫 | 看中两台洗衣机，参数差不多，不知道选哪台 | 参数表看不懂实际差异 |
| 预算卡位 | 3000 元买空调，想知道这个价位哪台性价比最高 | 不知道哪些参数值这个价 |
| 需求匹配 | 三口之家，想知道多大容量的冰箱合适 | 不知道自己的需求对应什么规格 |
| 升级换代 | 老家电要换，想知道新品比老款好在哪 | 不确定是否值得多花钱 |
| 品牌比较 | 海尔和美的同价位洗衣机，各有什么优劣 | 品牌宣传各说各好，缺乏客观对比 |

### 2.2 次要用户：前端产品经理（内部）

- 消费 AI 输出的结构化数据
- 需要数据格式稳定、字段含义明确
- 需要足够的信息粒度来设计不同的展示方案

## 3. 用户故事

### P0 - 核心功能（一期必须交付）

**US-01：同品类商品对比**

作为消费者，我希望选择同品类的两个或多个商品进行对比，以便了解它们在功能、参数、价格方面的实际差异。

验收标准见第 5 节。

**US-02：商品适配判断**

作为消费者，我希望输入我的使用场景（如家庭人数、使用频率、安装空间），以便知道某个商品是否适合我的需求。

验收标准见第 5 节。

**US-03：性价比评估**

作为消费者，我希望看到一个商品在同品类同价位段中的性价比定位，以便判断它是否值得购买。

验收标准见第 5 节。

### P1 - 重要功能（一期尽量交付）

**US-04：对比维度语义分组**

作为消费者，我希望对比结果按照我能理解的维度分组（如"洗净能力""节能表现""噪音控制"），而不是按原始参数表罗列，以便快速抓住关键差异。

验收标准：
- 每个品类有预定义的语义维度分组（如洗衣机：洗净能力、容量适配、节能表现、噪音控制、衣物保护、智能功能）
- 同一维度下的参数来自不同商品时，能正确对齐（如商品 A 的"BLDC 变频电机"和商品 B 的"直驱变频电机"归入同一维度）
- 不同品类的维度分组不同（洗衣机的维度和空调的维度不一样）

**US-05：差异摘要**

作为消费者，我希望看到两个商品之间最关键的 3-5 个差异点总结，以便不需要逐项比对就能快速决策。

验收标准：
- 差异点按对购买决策的影响程度排序，而不是按参数表顺序
- 每个差异点包含：维度名称、两个商品的具体值、对消费者的实际影响说明
- 差异说明用消费者能理解的语言，不用纯技术术语（如"噪音低 5 分贝，相当于从正常说话声降到轻声耳语"）

**US-06：不可比提示**

作为消费者，我希望在选择了不同品类商品（如洗衣机和微波炉）进行对比时，系统能告诉我这两个商品不适合直接对比，并说明原因，以便避免无意义的比较。

验收标准：
- 系统能识别跨品类对比请求并返回不可比标记
- 返回不可比原因（如"洗衣机和微波炉属于不同品类，功能完全不同，无法进行有意义的对比"）
- 对于近似品类（如洗衣机和洗烘一体机），标记为"部分可比"，并说明可比和不可比的部分

### P2 - 增强功能（二期或后续迭代）

**US-07：历史价格参考**

作为消费者，我希望看到商品的历史价格走势，以便判断当前价格是否合理。

验收标准：
- 展示近 30/90 天的价格区间（最高价、最低价、当前价）
- 标注当前价格在历史区间中的位置

**US-08：用户评价摘要**

作为消费者，我希望看到该商品真实用户的评价摘要（正面和负面），以便从其他用户的经验中获得参考。

验收标准：
- 提取 Top 3 正面和 Top 3 负面评价要点
- 每个要点标注支撑评价的数量（如"xx% 的用户提到了这一点"）

**US-09：替代商品推荐**

作为消费者，我希望在查看某个商品时，系统能推荐同品类中性价比更高或更适合我的替代选项，以便扩大选择范围。

验收标准：
- 推荐商品数量 2-3 个
- 每个推荐附带推荐理由（如"相同洗净能力，价格低 15%"）
- 推荐依据是 AI 语义理解的功能匹配度，不是简单的价格排序

## 4. 核心功能流程

### 4.1 商品对比流程

```
消费者在商品详情页或列表页触发对比
  -> 前端将商品 ID 列表发送给后端 API
  -> 后端从商品知识库获取商品结构化数据和非结构化描述
  -> Product Comparison Agent 执行以下步骤：
     1. 品类识别：判断商品是否属于同品类或近似品类
     2. 可比性判断：确定对比是否有意义，标记不可比/部分可比的情况
     3. 语义维度对齐：根据品类确定对比维度，将不同商品的参数映射到统一语义维度
     4. 参数解读：将技术参数转化为消费者可理解的功能描述
     5. 差异分析：识别关键差异点，按对购买决策的影响程度排序
     6. 性价比计算：基于功能覆盖度和价格做性价比定位
  -> 后端返回结构化 JSON 给前端
  -> 前端按自己的产品设计进行展示
```

### 4.2 适配判断流程

```
消费者在商品详情页触发"适不适合我"
  -> 前端将商品 ID 和用户场景参数发送给后端 API
     用户场景参数包括（前端按品类动态展示相关选项）：
     - 家庭人数
     - 使用频率
     - 安装空间（可选）
     - 预算范围（可选）
     - 特殊需求（如有婴幼儿、过敏体质等）
  -> 后端从商品知识库获取商品数据
  -> Product Comparison Agent 执行以下步骤：
     1. 需求解析：将用户场景翻译成具体的产品规格要求
     2. 匹配评估：将商品实际规格与用户需求逐项对比
     3. 适配判定：输出适配度评级和分维度评估
     4. 风险提示：标注可能不满足用户需求的维度
  -> 后端返回结构化 JSON 给前端
```

### 4.3 性价比评估流程

```
消费者在商品详情页触发性价比评估（或系统默认展示）
  -> 前端将商品 ID 发送给后端 API
  -> 后端从商品知识库获取同品类同价位段的商品集合
  -> Product Comparison Agent 执行以下步骤：
     1. 价位段划分：确定该商品所在价位段（如 2000-3000 元洗衣机）
     2. 功能基准线：统计该价位段商品的常见功能配置
     3. 功能溢出/缺失分析：标注该商品相比基准线多了什么、少了什么
     4. 性价比定位：给出该商品在价位段中的定位
  -> 后端返回结构化 JSON 给前端
```

## 5. P0 用户故事的验收标准

### US-01：同品类商品对比

**AC-01-1：基本对比功能**

- Given 消费者选择了同品类的 2-4 个商品
- When 发起对比请求
- Then 系统在 3 秒内返回结构化对比结果，包含：品类标识、可比性判断、至少 5 个对比维度、每个维度下各商品的参数值和功能解读

**AC-01-2：语义维度对齐**

- Given 两个洗衣机商品，商品 A 参数表写"BLDC 变频电机"，商品 B 参数表写"DD 直驱变频"
- When 发起对比请求
- Then 系统将两者归入同一个维度（如"电机类型"），并分别解读其对洗衣效果的影响，而不是把它们当作两个不同维度

**AC-01-3：跨品类拦截**

- Given 消费者选择了一台洗衣机和一台微波炉
- When 发起对比请求
- Then 系统返回不可比标记，不输出无意义的对比表

**AC-01-4：近似品类处理**

- Given 消费者选择了一台洗衣机和一台洗烘一体机
- When 发起对比请求
- Then 系统标记为"部分可比"，对洗涤功能做正常对比，对烘干功能标注"仅洗烘一体机具备"

**AC-01-5：缺失参数处理**

- Given 某个商品的商品知识库中缺少某个维度的数据
- When 发起对比请求
- Then 该维度标记为"数据不足"，不虚构参数值

### US-02：商品适配判断

**AC-02-1：基本适配判断**

- Given 消费者查看一台 8kg 洗衣机，并输入"三口之家，每天洗一次"
- When 发起适配判断请求
- Then 系统返回适配度评级（如"适合"/"基本适合"/"可能不足"/"不建议"），并给出容量适配的具体说明（如"8kg 容量对三口之家日常洗涤足够，可一次处理约 3-4 天的常规衣物"）

**AC-02-2：不满足项提示**

- Given 消费者查看一台无烘干功能的洗衣机，并输入"南方梅雨季节使用"
- When 发起适配判断请求
- Then 系统在适配结果中标注"烘干需求"维度为"不满足"，并说明原因

**AC-02-3：多维度评估**

- Given 消费者输入了场景参数
- When 发起适配判断请求
- Then 系统返回至少 3 个维度的评估（如容量、功能匹配、能耗），每个维度有独立的适配判定和说明

**AC-02-4：场景参数缺失时的降级**

- Given 消费者没有输入任何场景参数
- When 发起适配判断请求
- Then 系统返回通用适配说明（不基于具体场景的产品特点概述），不输出"非常适合您"这类无依据的判断

### US-03：性价比评估

**AC-03-1：价位段定位**

- Given 消费者查看一台 2599 元的洗衣机
- When 发起性价比评估请求
- Then 系统返回该商品所在价位段（如"2000-3000 元滚筒洗衣机"），以及该价位段的商品数量

**AC-03-2：功能基准线对比**

- Given 消费者查看一台商品
- When 发起性价比评估请求
- Then 系统返回该商品相比价位段基准线的功能溢出项和功能缺失项，每项附带说明

**AC-03-3：性价比定位**

- Given 消费者查看一台商品
- When 发起性价比评估请求
- Then 系统返回性价比定位（如"功能领先型"/"均衡型"/"基础实用型"），定位有明确依据而不是模糊的"高性价比"

**AC-03-4：价位段内无足够对比商品时的降级**

- Given 该价位段内平台只有 1 个商品（即自身）
- When 发起性价比评估请求
- Then 系统返回提示"该价位段商品数量不足，无法进行有效的性价比对比"，不虚构对比结果

### 6.4 关键字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `trace_id` | string | 是 | 全链路追踪 ID，用于审计和问题排查 |

| `risk_flags[].level` | enum | 是 | `warning` / `info`，warning 表示可能影响使用 |
| `positioning.type` | enum | 是 | `feature_leading` / `balanced` / `budget_practical` / `premium_niche` |
| `data_quality.confidence` | float | 是 | 0-1 之间，反映 AI 对本次输出的置信度 |
| `model_version` | string | 是 | AI 模型版本，用于追溯 |
| `knowledge_version` | string | 是 | 商品知识库版本，用于追溯 |

## 7. 一期边界

### 7.1 一期包含

| 项目 | 说明 |
| --- | --- |
| 同品类商品对比 | 支持 2-4 个同品类商品的结构化对比 |
| 商品适配判断 | 基于用户输入的场景参数判断商品适配度 |
| 性价比评估 | 基于同品类同价位段的功能基准线做定位 |
| 品类覆盖 | 一期试点品类：洗衣机、空调、冰箱（3 个品类） |
| 语义维度对齐 | AI 理解参数含义，将不同表述映射到统一维度 |
| 不可比拦截 | 跨品类对比时返回不可比标记 |
| 缺失数据标注 | 数据不足时明确标注，不虚构 |
| 结构化 JSON 输出 | 输出给前端，不直接面向消费者 |
| 数据追溯 | 每次输出附带 trace_id、model_version、knowledge_version |

### 7.2 一期不包含

| 项目 | 原因 |
| --- | --- |
| 对话式交互 | 一期只输出结构化数据，不做对话 AI |
| 跨平台比价 | 跨平台价格对比是 B 端工具场景，不在本需求范围 |
| 历史价格分析 | 需要价格数据积累，放到二期 |
| 用户评价分析 | 需要评论数据清洗和情感分析能力，放到二期 |
| 替代商品推荐 | 依赖较完善的品类商品覆盖度，一期先做对比和适配 |
| 全品类覆盖 | 一期只做 3 个试点品类，验证方法论后再扩展 |
| 个性化推荐 | 需要用户行为数据积累，一期不做 |
| 前端展示层 | 前端产品经理负责，本需求只定义数据结构 |
| 实时价格更新 | 一期使用商品知识库中的价格快照，不做实时价格追踪 |
| 线下门店场景 | 一期聚焦线上商城，不涉及线下门店导购场景（导购培训是独立场景） |

### 7.3 对商品知识底座的依赖

本需求的前提是商品知识底座具备以下能力：

- 每个商品有结构化参数数据（品类、型号、参数表）
- 每个商品有非结构化描述数据（卖点文案、详情页文字）
- 每个品类有预定义的语义维度映射规则（哪些参数归入哪个维度）
- 知识库有版本管理，输出结果可追溯到具体版本

如果知识底座未就绪，本功能的输出质量将直接下降。建议商品知识底座的建设与本功能同步推进，至少保证试点品类（洗衣机、空调、冰箱）的数据完备度。

### 7.4 与现有 Agent Matrix 的关系

本需求新增一个 Agent：**Product Comparison Agent**。

- 它复用现有的 Knowledge RAG Service 获取商品知识
- 它复用现有的 Offer Normalization Engine 做价格标准化
- 它不依赖 OpenClaw 采集（数据来源是平台内部商品库）
- 它的输出经过 Risk & QA Agent 做质量校验
- 它受 Orchestrator Agent 编排，通过统一 API Gateway 对外

建议在架构中的位置：

```
消费者前端
  -> API Gateway
  -> Orchestrator Agent
  -> Product Comparison Agent
     -> Knowledge RAG Service（获取商品知识）
     -> Offer Normalization Engine（价格标准化）
  -> Risk & QA Agent（质量校验）
  -> 返回结构化 JSON
```
