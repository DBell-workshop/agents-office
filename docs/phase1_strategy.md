# 一期方案：导购培训 + 跨平台比价

## 1. 一期目标

一期只聚焦两个场景，不追求大而全：

- 场景 A：线下门店导购培训
- 场景 B：跨平台比价与福利机制比较

目标不是做一个“万能聊天 Agent”，而是做一套可复用的 AI 能力底座，并先在两个高价值、可验证场景里打穿闭环。

一期结束时，希望拿到四个结果：

- 导购培训形成完整闭环：学、听、练、评、改
- 跨平台比价形成完整闭环：采、比、析、审、用
- 共用能力沉淀成 API，后续其他产品团队可复用
- 有明确试点指标，能决定是否进入二期扩展

## 2. 为什么一期只做这两个场景

这两个场景兼顾了价值、可落地性和能力复用度。

### 2.1 导购培训

业务价值：

- 直接作用于线下转化质量
- 可沉淀商品知识、话术模板、训练标准
- 容易做试点和效果对比

技术价值：

- 能沉淀 RAG、TTS、ASR、评分引擎、内容版本管理
- 为后续客服培训、直播培训、导购协同提供底座

### 2.2 跨平台比价

业务价值：

- 直接支持运营决策和价格策略判断
- 能为后续用户侧商品比较提供能力基础
- 容易证明效率收益

技术价值：

- 能沉淀跨平台商品对齐、价格与福利标准化、证据展示能力
- 为后续竞品监控、用户比较推荐提供数据与规则底座

## 3. 一期总体原则

- Agent 只做高价值推理与生成，不承担所有确定性计算
- 高稳定任务优先走工作流和规则，不走自由发挥
- 所有输出必须可追溯、可审计、可回看证据
- 高风险能力先做辅助，不直接进入刚性考核和自动决策
- OpenClaw 只作为一期比价的浏览器采集适配层，不作为长期数据平台

## 4. 一期 Agent Matrix

一期建议只保留 6 个核心 Agent。

| Agent | 服务对象 | 核心职责 | 输入 | 输出 | 风险等级 |
| --- | --- | --- | --- | --- | --- |
| Orchestrator Agent | 全场景 | 意图识别、流程路由、上下文装配、权限控制 | 用户请求、角色、上下文 | 工作流、结果聚合 | 中 |
| Sales Coach Agent | 导购、培训负责人 | 生成卖点、话术、模拟问答、训练内容 | 商品知识、课程要求、门店场景 | 话术稿、训练卡、TTS 文本 | 中 |
| Pitch Evaluator Agent | 导购、店长 | 录音转写、事实核验、停顿分析、评分建议 | 录音、标准话术、商品事实、评分标准 | 分维度评分、错误点、改进建议 | 高 |
| Price Intelligence Agent | 运营 | 采集后数据对齐、价格与福利对比、差异解释、经营建议 | 商品 ID、平台页面数据、规则 | 对比结果、建议动作、证据 | 高 |
| Risk and QA Agent | 全场景 | 审核事实错误、违规表述、敏感风险 | 所有 AI 输出、规则、商品事实 | 审核结果、修正建议、拦截记录 | 高 |
| Experiment Analyst Agent | 实验室、管理层 | 汇总试点指标、分析 ROI、识别问题 | 埋点、使用日志、试点数据 | 周报、复盘、扩展建议 | 低 |

## 5. 共用能力层

一期的重点不是把 Agent 做多，而是把底层能力做对。

| 能力 | 作用 | 一期要求 |
| --- | --- | --- |
| 商品知识底座 | 支撑话术、评分、比价、解释 | 至少覆盖试点类目核心属性 |
| 内容版本管理 | 支撑运营审核和追责 | 每个话术、课程、评分标准可追溯 |
| RAG 检索 | 支撑商品知识召回 | 输出必须附带知识来源 |
| TTS | 支撑听练 | 自然语音、可切换音色、时长可控 |
| ASR | 支撑录音点评 | 支持口语转写和时间戳 |
| 流畅度分析 | 支撑停顿、重复、口头禅识别 | 至少支持录音级别分析 |
| 评分引擎 | 支撑可解释评分 | 输出分维度分数与证据 |
| 平台采集适配层 | 支撑比价原始数据采集 | 一期使用 OpenClaw |
| 价格和福利标准化 | 支撑对比结果统一口径 | 输出标准字段和计算结果 |
| 审计和日志 | 支撑治理和回溯 | 模型、Prompt、知识版本、人工修订全记录 |

## 6. 场景 A：导购培训闭环

### 6.1 目标用户

- 一线导购
- 店长
- 培训运营

### 6.2 一期核心流程

1. 培训运营选择商品，配置课程目标和必讲卖点
2. Sales Coach Agent 基于商品知识生成训练卡、话术稿、模拟问答
3. 培训运营审核和修订内容，形成正式版本
4. 导购在 App 中查看内容并收听 TTS 语音示范
5. 导购进行录音练习
6. Pitch Evaluator Agent 做转写、事实核验、流畅度分析和评分
7. 系统返回逐条问题、时间点、修改建议和复训建议
8. 店长查看导购训练结果，做人工复核

### 6.3 一期功能范围

- 商品学习卡
- AI 话术生成
- TTS 播放
- 录音上传
- 自动转写
- 卖点覆盖分析
- 事实错误标注
- 停顿和流畅度分析
- 分维度评分
- 店长复核

### 6.4 一期先不做

- 视频动作和表情分析
- AI 评分直接影响绩效
- 完全自动化淘汰或处罚
- 全品类铺开

### 6.5 建议评分维度

| 维度 | 权重建议 | 说明 |
| --- | ---: | --- |
| 事实准确度 | 30% | 参数、卖点、权益有没有讲错 |
| 卖点覆盖率 | 20% | 必讲卖点覆盖是否完整 |
| 表达流畅度 | 15% | 停顿、重复、卡壳、口头禅 |
| 话术自然度 | 10% | 是否像真人、是否过于机械 |
| 场景匹配度 | 15% | 是否按用户需求组织表达 |
| 合规性 | 10% | 是否夸大、绝对化、敏感表述 |

要求：

- 每个维度都必须给证据
- 每个错误都必须定位到句子或时间点
- 总分只能作为辅导依据，一期不直接入绩效

## 7. 场景 B：跨平台比价闭环

### 7.1 目标用户

- 价格运营
- 类目运营
- 竞品分析人员

### 7.2 一期核心流程

1. 运营输入我方商品或竞品链接
2. 系统选择采集目标平台和页面
3. OpenClaw 驱动浏览器访问页面并抽取结构化字段
4. 平台采集适配层把原始页面信息转成标准字段
5. Price Intelligence Agent 识别同款或近似款，计算价格与福利差异
6. Risk and QA Agent 校验结果是否有字段缺失、口径错误、可比性问题
7. 系统输出结构化对比表、差异总结、建议动作和证据截图

### 7.3 一期功能范围

- 商品链接录入
- 页面字段采集
- 同款和近似款判断
- 到手价、券后价、满减、赠品、会员权益、分期、以旧换新等标准化
- 对比表展示
- 差异解释
- 异常标记
- 证据留存

### 7.4 一期先不做

- 全平台全量实时监控
- 自动化价格决策闭环
- 未经人工复核的经营动作下发
- 依赖页面抓取的强 SLA 服务承诺

## 8. OpenClaw 在比价场景中的定位

由于一期没有稳定数据接口，OpenClaw 可以作为浏览器采集适配层，用于驱动浏览器访问平台页面、读取页面内容、截图留证、辅助字段抽取。

但要明确它在系统中的角色：

- 它是采集入口，不是比价引擎
- 它是实验室和试点阶段能力，不是长期唯一数据底座
- 它负责“看见页面并抓回证据”，不负责最终业务结论

### 8.1 推荐架构位置

```text
运营输入商品/链接
  -> Orchestrator Agent
  -> OpenClaw Browser Runner
  -> 页面字段抽取与截图留证
  -> 标准化与清洗
  -> Price Intelligence Agent
  -> Risk and QA Agent
  -> 对比报告与经营建议
```

### 8.2 一期为什么适合用 OpenClaw

- 没有现成数据接口时，可以较快拉起浏览器采集能力
- 可以保留页面截图、页面文本、操作轨迹，便于追溯
- 可以按任务访问指定页面，而不是一次性建设复杂爬虫体系
- 实验室场景允许以“半自动、可复核”的方式快速验证需求

### 8.3 必须接受的限制

- 页面结构变化会导致采集规则失效
- 登录状态、反爬、验证码会影响稳定性
- 采集时延高于接口化数据
- 不适合承诺强实时、强 SLA、全量覆盖

### 8.4 正确的工程用法

- 每个平台单独配置采集模板和字段映射
- 把 OpenClaw 输出与业务标准字段隔离
- 每次采集保留原始快照和截图
- 对无法判断的字段显式标记为不确定，不允许模型猜
- 高价值页面优先人工抽样复核

### 8.5 后续替代路径

二期如果效果成立，应逐步把 OpenClaw 退出核心依赖，转向更稳定的数据方式：

- 官方或合作接口
- 合规的结构化数据通道
- 平台或品牌侧数据合作
- 自建更稳定的采集和解析服务

## 9. 一期技术架构建议

### 9.1 逻辑分层

| 层级 | 职责 |
| --- | --- |
| 体验层 | 导购 App、运营后台、试点报表 |
| 编排层 | Orchestrator Agent、工作流、权限控制 |
| Agent 层 | Sales Coach、Pitch Evaluator、Price Intelligence、Risk and QA、Experiment Analyst |
| 能力层 | RAG、TTS、ASR、评分引擎、采集适配层、价格规则引擎 |
| 数据层 | 商品主数据、课程版本、训练记录、采集记录、对比结果、日志审计 |

### 9.2 API 方向

建议一期就按能力接口建设，而不是按页面写死：

- `generate_training_content`
- `synthesize_training_audio`
- `evaluate_sales_pitch`
- `collect_competitor_page`
- `normalize_offer_fields`
- `compare_offers`
- `audit_ai_output`
- `get_trial_metrics`

## 11. 一期验收指标

### 11.1 导购培训

- 训练完成率
- 复训率
- AI 与店长评分一致性
- 人均训练准备时长下降
- 导购卖点覆盖率提升
- 店长对点评结果认可度

### 11.2 跨平台比价

- 单次分析耗时下降
- 标准字段抽取成功率
- 同款识别准确率
- 运营采纳率
- 采集失败率
- 人工复核通过率

## 12. 风险和治理

一期最容易踩坑的是三个点：

- 把 OpenClaw 当长期数据底座
- 把 AI 评分直接绑定绩效
- 把模型生成结果当确定事实

所以要定三条红线：

- 采集不稳定时，系统必须降级为人工复核，不允许输出伪确定结论
- AI 评分一期只做辅导，不直接用于处罚和奖金扣减
- 所有核心结果必须保留证据、版本和审核记录

## 13. 领导汇报的一句话

一期建议聚焦“导购培训”和“跨平台比价”两个高价值场景，用可编排的 Agent 能力平台做闭环验证；其中比价场景在没有数据接口的前提下，以 OpenClaw 作为实验室阶段的浏览器采集适配层，先跑通“采集、标准化、对比、审核、使用”的业务闭环，再决定二期是否建设更稳定的数据通道。
