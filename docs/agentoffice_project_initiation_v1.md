# AgentOffice 项目立项申请（初稿）

## 一、项目概述

项目名称：AgentOffice

项目定位：建设公司 AI 实验室中的原型验证平台，以 Agent 和可复用 AI 服务能力为核心，面向业务场景快速完成原型研发、试用验证与生产适配评估。

本项目不是直接建设集团正式生产系统，而是建立一套可控、隔离、低风险的创新验证机制。通过实验室先行的方式，缩短从想法到可验证原型的周期，再决定是否进入正式生产改造。

## 二、为什么现在要做

当前公司在数据安全、系统接入、网络环境和审批流程上要求严格，这对正式生产是必要的，但不适合创新早期高频试错。AI 场景尤其如此，若一开始就按正式生产标准推进，往往会在价值尚未验证之前消耗大量时间和资源。

因此，需要在正式生产体系之外，建立一个边界清晰的 AI 实验室研发路径：

- 本地实验室研究开发原型机
- 小范围邀请公司内外部用户试用
- 适配正式生产环境改造
- 在集团正式生产环境上线

这个路径的核心价值是：先验证业务价值与体验上限，再投入生产化成本。

## 三、一期项目范围与场景

一期聚焦 3 个高价值场景，不做大而全的平台铺设。

| 场景 | 目标用户 | 价值定位 | 一期交付形式 |
| --- | --- | --- | --- |
| 导购培训 | 门店导购、培训运营、店长 | 提升导购对商品信息、卖点和讲解流程的掌握程度，形成练习、点评、改进闭环 | Web 原型 + 后端 Agent 服务 |
| 跨平台比价 | 运营人员 | 支持低频但高价值的价格、福利、促销机制比较，辅助运营判断 | 内部工具型能力 |
| 平台内商品对比 | 消费者，前端产品团队 | 为客户端 App 提供商品差异分析与推荐能力，提高用户决策效率 | API 服务，不在一期内建设 C 端页面 |

一期不做的内容：

- 不做集团正式生产环境接入
- 不做完整 C 端展示页面
- 不做复杂权限体系和企业级审计改造
- 不承诺正式生产级 SLA

## 四、项目要解决的核心问题

### 1. 导购培训

- 商品信息复杂，卖点理解和表达不稳定
- 新品、活动、权益更新快，培训成本高
- 缺少标准化练习和客观反馈闭环

### 2. 跨平台比价

- 价格、优惠券、赠品、会员权益等口径复杂
- 人工比较效率低，且结论依赖个人经验
- 低频场景不适合建设重型系统，但对经营判断价值高

### 3. 平台内商品对比

- 同品类商品属性复杂，消费者难以理解差异
- 家电和耗材类商品参数异构，前端难以稳定封装
- 需要以 API 的形式给客户端提供可解释的商品比较能力

## 五、项目价值

### 1. 业务价值

- 缩短 AI 场景从想法到原型验证的周期
- 提高导购培训效率与训练质量
- 提升运营在关键时点的比价分析效率
- 为客户端提供可复用的商品对比与推荐能力

### 2. 组织价值

- 建立公司内部 AI 创新验证机制
- 将“先原型、后生产化”固化为标准路径
- 避免在价值未验证前进入重型流程

### 3. 技术价值

- 沉淀 Agent 编排、模型调用、任务管理、日志追踪等底层能力
- 为后续正式生产系统输出可复用的服务边界和接口设计
- 积累高质量商品理解、卖点抽取、属性归一化等样本资产

## 六、实施路径

项目按 4 个阶段推进：

| 阶段 | 目标 | 说明 |
| --- | --- | --- |
| 阶段 1：实验室原型研发 | 跑通核心业务闭环 | 在隔离环境中快速实现原型机，优先验证体验和效果 |
| 阶段 2：小范围试用 | 验证真实可用性 | 邀请公司内外部少量用户试用，收集反馈和指标 |
| 阶段 3：生产适配改造 | 解决合规、网络、权限、审计问题 | 针对确定有价值的能力做正式化改造 |
| 阶段 4：集团上线 | 接入正式生产环境 | 按集团规范完成上线和运营 |

该路径的关键原则：

- 原型期优先验证“业务价值”和“产品体验”
- 正式版再验证“合规、成本、稳定性和替代性”

## 七、模型策略与技术路线

### 1. 为什么原型期优先使用国际一流大模型

原型期的核心目标，不是控制单次推理成本，而是尽快验证场景是否成立、体验是否可接受、能力边界在哪里。

国际领先模型在以下方面更适合作为原型期底座：

- 复杂推理稳定性更高
- 多模态能力更成熟，适合处理文本、图片、语音等混合输入
- 结构化输出、工具调用和 Agent 编排生态更成熟
- API 和评测工具链更完善，便于快速试错
- 更容易验证“场景上限”，避免把模型短板误判成场景无价值

### 2. 为什么不直接把国产模型作为原型唯一底座

项目并不是排斥国产模型，而是区分“原型验证阶段”和“生产落地阶段”的目标不同：

- 原型阶段优先看效果上限和研发效率
- 正式生产阶段优先看合规、可控、成本和替代性

如果在原型阶段就只使用效果尚未充分验证的替代方案，可能会导致：

- 场景价值被低估
- 原型体验不达标
- 无法准确判断业务是否值得投入

因此，建议策略是：

- 原型阶段：优先使用国际领先模型
- 生产评估阶段：引入国产模型或开源模型做 A/B 测试
- 正式上线前：根据效果、成本、时延、合规结果决定是否替代

### 3. 是否可以用一流模型“带教”我们自己的模型

可以，但建议用更稳妥的方式理解这件事：

- 用领先模型对自有商品数据做结构化标注和高质量样本生成
- 将沉淀下来的自有样本用于任务级微调或蒸馏
- 对高频、低风险任务逐步切换到国产模型或开源模型
- 对高复杂度任务继续保留领先模型作为兜底能力

需要特别注意：

- 是否允许使用某家模型输出去训练其他模型，取决于供应商条款和法务判断
- 因此原型期更稳妥的做法是：先做评测、样本沉淀和任务级验证，不急于训练通用模型

## 八、资源与预算申请

本项目当前为实验室原型阶段，不申请人力预算，仅申请必要的实验资源预算。

### 1. 资源清单

| 资源项 | 用途 | 申请说明 |
| --- | --- | --- |
| 海外 VPS | 运行隔离实验环境、代理服务、必要的远程采集节点 | 建议 1 到 2 台轻量节点 |
| VPN / 安全访问链路 | 保障实验环境访问和网络隔离 | 可采用自建 WireGuard 或等价方案 |
| 国际领先模型 API | 支撑原型期多模型对比和高质量能力验证 | 建议至少 2 家供应商并行 |
| 独立 Mac 采集设备 | 用于特定浏览器采集或 OpenClaw 远程节点 | 视场景按需采购或复用现有设备 |

### 2. 月度预算建议（不含人力）

| 预算项 | 建议月度预算 | 说明 |
| --- | --- | --- |
| 海外 VPS | 20 到 80 美元 / 月 | 轻量实验节点，1 到 2 台 |
| VPN / 访问控制 | 0 到 30 美元 / 月 | 若采用自建方式，软件成本可接近 0 |
| 大模型 API 充值 | 500 到 1500 美元 / 月 | 用于原型研发、评测、A/B 对比和演示 |
| 预备弹性预算 | 100 到 300 美元 / 月 | 用于临时扩容、额外 API 调用或试用活动 |

建议实验室阶段先申请 3 个月预算，总额按月度预算区间滚动管理。

### 3. 预算说明

- 海外 VPS 预算以主流云厂商轻量节点为参考，原型期以最小可用配置为准
- API 预算不是单一模型费用，而是多家领先模型的探索和对比预算池
- 若验证结果成立，后续可以通过国产替代或开源替代显著降低正式生产成本

## 十一、立项建议

建议将 AgentOffice 定位为“AI 实验室原型验证项目”，而不是直接定义为正式生产系统建设项目。

项目当前最合理的决策方式不是一次性投入重资源建设完整平台，而是先以低成本、高效率方式完成以下验证：

- 场景是否有真实业务价值
- 用户是否愿意使用
- 当前领先模型能否支撑体验要求
- 国产或开源替代在正式版中是否可行

## 附：一句话总结

AgentOffice 不是直接建设正式生产系统，而是建设公司 AI 实验室中的原型验证机制。通过先在隔离环境中用国际领先模型快速验证导购培训、跨平台比价和平台内商品对比 3 个场景的业务价值，再决定是否做生产改造，并通过 A/B 测试验证国产或开源模型的替代可行性。
