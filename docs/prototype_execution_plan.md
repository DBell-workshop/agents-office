# 原型执行计划：由我直接搭建一期 Agent 原型

## 1. 目标

这个计划按“实验室原型”方式设计，不按正式产研协作方式设计。

目标是由我直接完成一套可演示、可操作、可扩展的一期原型，覆盖两个场景：

- 导购培训
- 跨平台比价

原型目标不是正式生产上线，而是：

- 跑通关键业务闭环
- 让领导和业务看懂能力边界
- 让后续研发看到可落地的系统形态
- 为二期工程化提供真实原型和接口基础

## 2. 我会怎么做

我不会先做“很多 Agent 互相协作”的复杂系统。

我会按下面这个顺序来做：

1. 先搭一个统一原型底座
2. 先做一个调度层和两个工作流
3. 再把每条工作流封装成对应 Agent
4. 最后补演示页面、日志、审计和配置化

也就是说，第一版不是 Agent 社会，而是：

- 一个 `Orchestrator`
- 一个 `Training Workflow`
- 一个 `Comparison Workflow`
- 一组独立能力服务

这样做的好处是：

- 更快出效果
- 更容易调试
- 更容易解释
- 更容易后续交给研发重构

## 3. 我建议的原型技术路线

为了让我单兵快速交付，我建议采用“前后端分层但尽量轻量”的结构：

### 3.1 前端

- 一个轻量 Web 控制台
- 包含两个入口：
  - 导购培训台
  - 跨平台比价台

前端原型目标：

- 能让业务操作
- 能看见 Agent 输出
- 能看见证据和日志

### 3.2 后端

- 一个 API 服务
- 一个工作流/调度层
- 一组能力服务封装

后端原型目标：

- 跑通两个场景
- 支持异步任务
- 记录审计和过程结果

### 3.3 外部能力接入

- LLM：生成话术、总结差异、解释建议
- TTS：生成示范音频
- ASR：转写导购录音
- OpenClaw：采集竞品页面

### 3.4 存储

原型阶段只需要最小可用存储：

- 商品知识
- 课程配置和版本
- 练习记录和评分报告
- 比价任务和结果
- 页面截图和采集快照
- 审计日志

## 4. 一期我要实际交付什么

### 4.1 导购培训原型

- 商品课程配置页
- AI 训练内容生成
- 运营审核发布
- TTS 语音示范
- 导购录音上传
- 自动转写和评分
- 结果页展示问题、证据、建议

### 4.2 跨平台比价原型

- 输入自家商品和竞品链接
- 调用 OpenClaw 采集页面
- 标准化价格和福利字段
- 输出结构化对比结果
- 附带页面截图和采集时间
- 显示异常和不确定项

### 4.3 共用平台能力

- 调度层
- 能力 API
- 配置中心雏形
- 任务状态管理
- 审计日志

## 5. 具体实施阶段

## Phase 0：冻结范围和准备环境

目标：

- 明确一期只做两个场景
- 确认商品试点范围
- 确认可用账号、Key、OpenClaw 接入条件

我会产出：

- 原型目录结构
- 配置模板
- 环境变量清单
- 基础数据样本清单

完成标准：

- 能确定 1 到 2 个试点类目
- 能拿到最小商品资料
- 能调用模型和 OpenClaw

## Phase 1：搭原型骨架

目标：

- 搭建前端壳子
- 搭建后端 API 服务
- 搭建任务模型和日志模型

我会先做：

- 项目脚手架
- API 路由
- 任务状态机
- 文件存储结构
- 审计日志中间件

完成标准：

- 页面能打开
- API 能跑
- 能创建任务并看到状态变化

## Phase 2：先打通导购培训链路

目标：

- 跑通从课程配置到评分反馈的最短闭环

我会做的模块：

- 商品知识录入/读取
- 训练内容生成
- 内容版本管理
- TTS 生成
- 录音上传
- ASR 转写
- 评分引擎
- 报告页

完成标准：

- 一个商品可完整完成“学、听、练、评、改”流程
- 报告里至少能识别事实错误、漏讲卖点、流畅度问题

## Phase 3：再打通跨平台比价链路

目标：

- 跑通从链接录入到比价报告输出的闭环

我会做的模块：

- 比价任务创建
- OpenClaw 采集适配
- 平台字段映射模板
- 价格和福利标准化
- 同款/近似款判断
- 差异解释
- 结果页与证据页

完成标准：

- 能对至少两个平台做一次完整对比
- 报告能展示截图、原始字段、标准字段、建议结论

## Phase 4：补调度层和治理层

目标：

- 把两个场景从“独立功能”收敛成统一能力平台

我会做的模块：

- Orchestrator
- 统一任务中心
- 统一审计
- 统一错误处理
- 统一配置管理

完成标准：

- 两条工作流都能走统一入口
- 任意任务都能按 trace_id 回看过程

## Phase 5：补演示能力和可视化

目标：

- 让业务和领导一眼看懂

我会做的模块：

- 首页和场景入口
- Demo 数据
- 指标看板
- 任务回放页
- 关键日志展示

完成标准：

- 非技术角色能在 10 分钟内看懂两个场景怎么跑
- 可以现场演示完整流程

## Phase 6：收尾和交付

目标：

- 形成可交接原型
- 为后续研发工程化提供明确基础

我会整理：

- README
- 环境配置
- API 文档
- 数据结构说明
- 已知风险清单
- 二期改造建议

完成标准：

- 新接手的人能把原型跑起来
- 研发能看懂边界、模块和接口

## 6. 我建议的开发顺序

如果由我单独完成，我会严格按这个顺序推进：

1. 原型骨架
2. 导购培训闭环
3. 跨平台比价闭环
4. 调度层统一
5. 治理和日志
6. Demo 和文档

原因很简单：

- 导购培训链路更可控，先打通能快速建立骨架
- 比价依赖 OpenClaw 和页面采集，不确定性更高，放第二条线更稳
- 调度层必须建立在工作流已经成型之后，否则会空转

## 7. 时间计划

如果让我连续推进，我建议按 4 到 6 周做第一版可演示原型。

### 第 1 周

- 搭骨架
- 配环境
- 建任务系统
- 建存储结构

### 第 2 周

- 完成导购培训内容生成
- 接 TTS
- 接录音上传和 ASR

### 第 3 周

- 完成评分引擎和报告页
- 把导购培训链路跑通

### 第 4 周

- 接 OpenClaw
- 做字段映射和标准化
- 跑通比价链路

### 第 5 周

- 做统一调度层
- 做统一日志和审计
- 做异常处理和降级

### 第 6 周

- 做 Demo 打磨
- 做测试样例
- 做交付文档

如果中间 OpenClaw 接入很顺利，4 周可以出第一版。

## 8. 我需要你提供的最小输入

如果你让我直接开工，我需要你给我下面这些最小输入：

- 试点商品范围，先给 3 到 10 个商品
- 导购培训要覆盖的 1 到 2 个类目
- 跨平台比价要先比哪几个平台
- 你可用的模型 API Key
- OpenClaw 的可用方式和账号条件

如果你现在没有准备齐，我也可以先用 mock 数据和假任务把骨架搭起来。

## 9. 我不会一开始做的东西

为了保证原型尽快出来，我会明确砍掉这些：

- 视频分析
- 绩效系统联动
- 多角色复杂权限体系
- 多租户设计
- 大规模并发优化
- 自动定价
- 真正意义上的多 Agent 自主协商

## 10. 风险和应对

### 风险 1：OpenClaw 采集不稳定

应对：

- 先只支持少量页面模板
- 强制保留截图和快照
- 采集失败直接降级为人工复核

### 风险 2：商品知识不完整

应对：

- 先限制试点商品数量
- 先人工整理知识样本
- 所有生成结果都挂证据

### 风险 3：评分不服众

应对：

- 一期只做辅导
- 分维度输出
- 必须带证据和具体建议

## 11. 我建议你现在怎么让我开始

最有效的方式不是继续空谈，而是直接进入原型建设。

我建议下一步按这个顺序：

1. 我先搭项目骨架
2. 我先实现导购培训 MVP
3. 我再接跨平台比价 MVP
4. 我最后统一调度和日志

## 12. 开工前的 4 个确认项

如果你同意我直接做，我建议你只确认这 4 件事：

1. 一期技术上就按原型优先，不要求正式生产规范
2. 先做 Web 原型，不先做原生 App
3. 先用 mock 商品数据加少量真实数据混合推进
4. 比价场景允许先用 OpenClaw 做实验室采集适配

这 4 件事你点头，我就可以直接开始搭第一版。
